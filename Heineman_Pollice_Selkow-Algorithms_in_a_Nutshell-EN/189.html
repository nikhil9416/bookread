At alternating levels, then, the program selects a move that maximizes score(state, initial), but when the opponent is making its move, the program assumes the opponent will select a move that mini‐ mizes score(state, initial). Of course, the program can look ahead only a finite number of moves because the game tree is potentially infinite. The number of moves chosen to look ahead is called the ply. The trade-off is to select a suitable ply that leads to a search explora‐ tion that completes in reasonable time. The following pseudocode illustrates the Minimax algorithm. Minimax Summary Best, Average, Worst: O(b ply ) bestmove (s, player, opponent) original = player [move, score] = minimax (s, ply, player, opponent) return move end minimax (s, ply, player, opponent) best = [null, null] if ply is 0 or there are no valid moves then score = evaluate s for original player return [null, score] foreach valid move m for player in state s do execute move m on s [move, score] = minimax(s, ply-1, opponent, player) undo move m on s if player is original then if score > best.score then best = [m, score] else if score < best.score then best = [m, score] return best end Remember original player since evaluation is always from that player’s perspec‐ tive. If no more moves remain, player might have won (or lost), which is equivalent to reaching target ply depth. With each recursive call, swap player and opponent to reflect alternating turns.
