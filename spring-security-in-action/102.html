The root of the muddy aspect of the pre- vious code example is a mix of two responsibilities. While it’s true that you need both in the application, in this case, nobody says that you have to put these into the same class. Let’s try to separate those by defining a separate class called SecurityUser , which decorates the User class. As listing 3.10 shows, the SecurityUser class imple- ments the UserDetails contract and uses that to plug our user into the Spring Secu- rity architecture.
