In the examples presented in this chapter, you find the authori- zation rules applied for methods of the service classes. But you can apply authorization rules with global method security in any part of your applica- tion: repositories, managers, proxies, and so on. 16.3 Applying postauthorization Now say you want to allow a call to a method, but in certain circumstances, you want to make sure the caller doesn’t receive the returned value. When we want to apply an authorization rule that is verified after the call of a method, we use postauthorization. It may sound a little bit awkward at the beginning: why would someone be able to exe- cute the code but not get the result? Well, it’s not about the method itself, but imagine this method retrieves some data from a data source, say a web service or a database. You can be confident about what your method does, but you can’t bet on the third party your method calls. So you allow the method to execute, but you validate what it returns and, if it doesn’t meet the criteria, you don’t let the caller access the return value. To apply postauthorization rules with Spring Security, we use the @Post- Authorize annotation, which is similar to @PreAuthorize, discussed in section 16.2. The annotation receives as a value the SpEL defining an authorization rule.
